# Choosing a technique — by subject and by goal

Pick from two angles: what the **subject** rewards, and what the **goal** is right now. The goal often wins — "memorize the 30 bones" and "understand why the reaction happens" need different methods even in the same class.

## By goal

- **Understand it** → Feynman, self-explanation, elaborative interrogation, concrete examples, dual coding.
- **Memorize it** → active recall + spaced repetition (flashcards / Anki), the Blurt, dual coding for visual facts.
- **Practice-apply it** (solve problems) → worked examples + self-explanation, then interleaved practice, plus an error log.
- **Prep for a test** → past papers under timed conditions, the Blurt, mixed / interleaved sets, and spaced reviews in the run-up.

## By subject

Use these as starting points, then adapt to the goal and the learner.

- **Math** — practice problems, interleaving, worked-example-then-redo, self-explanation of each step, Feynman for proofs, and an **error log** (every mistake, why it happened, and the fix). *Avoid* flashcards and rereading — math is learned by *doing*. Don't solve problems for them; coach the next step.
- **Sciences** — labeled redrawing (dual coding), terminology flashcards (spaced), "why does this happen?" (elaborative interrogation), past papers, concept maps. A strong stack: Cornell notes → active recall → concept map.
- **History** — timelines, cause-and-effect webs, date/event recall, practice essays, perspective cards (the same event from different viewpoints). Stack: Cornell → Blurt → practice essays.
- **English / Literature** — quote banks, theme and character diagrams, essay planning, reading aloud, argument mapping. Analyze *as* you read; avoid passive rereading.
- **Foreign languages** — Anki for vocab (spaced), shadowing (repeat aloud over audio), translating without looking, writing sentences in context, immersive listening. Frequent retrieval wins.
- **Computer science** — write code from scratch (no copying), rubber-duck debugging, hand-trace logic on paper, Feynman for algorithms. *Avoid* watching tutorials without coding along.
- **Geography / social studies** — label maps from memory, case-study cards, cause-effect diagrams, statistics recall.
- **Music / arts** — daily spaced practice (consistency beats marathon sessions), record-and-critique, theory recall, section-by-section mastery.

## Studying in multiple formats (not "learning styles")

Matching study to a fixed personal "style" doesn't improve learning — that idea has repeatedly failed in controlled studies. What *does* work: **match the format to the content** (a map wants to be seen, a poem heard, a procedure done), and **stack multiple formats** for anything important, because more encodings mean more ways to retrieve it later (that's dual coding). Treat these as a menu, not a personality type:

- **Visual** (structure / relationships) — mind and concept maps, color-coded-by-category notes, flowcharts, cover-and-redraw.
- **Auditory** (language / argument) — read aloud, teach it / record and replay, study groups, spoken Feynman.
- **Hands-on** (procedures, or when sitting still stalls) — walk-and-review, build models, whiteboard standing up, study games. (Handwriting helps because its slowness forces summarizing — depth of processing — not because someone is a "kinesthetic type.")
- **Text** (when precise wording or a written exam matters) — Cornell, rewrite in own words, written Feynman, self-built glossaries, timed essays.
- **The evidence-backed stack:** draw it → say it → write an example → teach it.

## When something isn't working

Diagnose by what they can produce without notes. If a method stalls, swap it (see the end of `techniques.md`). Change the technique, not the student's verdict on themselves.
